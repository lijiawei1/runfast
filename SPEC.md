# 夺A快跑 - SPEC（v0.10）
*更新日期：2026-06-23*

---

## 牌序规则
globalOrder = rank * 4 + suit
- rank: A=0, 2=1, 3=2, ..., 10=9, J=10, Q=11, K=12
- suit: ♦=0, ♣=1, ♥=2, ♠=3

排序：♦A=0 < ♣A=1 < ♥A=2 < ♠A=3 < ♦2=4 < ...

---

## 截断牌堆（5人局）
总牌数 = 人数 × 5 = 25张
只取 globalOrder 0~24 的牌
最大牌：♦7（order=24）

---

## 牌型（仅4种）
- 单张（single）
- 对子（pair）：2张同rank
- 三条（triple）：3张同rank
- 四条（quad）：4张同rank

---

## 比较规则
1. 牌型层级：quad > triple > pair > single
2. 同牌型比 rank
3. 同牌型同 rank：比最高 suit（♦<♣<♥<♠）

---

## 出牌规则
- 首出：持♦A者（或抢A后★）必须且只需"手牌包含♦A"
- 圈内出牌：后续玩家必须出同牌型且更大的牌（同牌型，先比rank，同rank比top_suit），不能出不同牌型
- Pass（不出）：当玩家管不上桌上的牌时，选择不出，不改变trick，只将turn移到下家
- 一圈无人管：当turn回到最初出牌人（starter）时，清空trick，出牌人继续自由出牌（无接风概念）

### 特殊约束：下家只剩一张牌
- **触发条件**：当前出牌人（turn）的顺时针下家（即 `(turn + 1) % 5`）手牌数为1时
- **约束内容**：
  - 若当前出牌人选择出**单张**：必须出自己手牌中**最大的单张**（按 globalOrder = rank×4+suit 最大）
  - 若当前出牌人选择出**非单张牌型**（对子/三条/四条）：不受约束，可以自由选择
- **生效场景**：
  - 自由出牌（trick=None）时
  - 接力出牌（trick!=None）时
  - 一圈无人管后继续出牌时
- **不生效场景**：
  - 下家手牌数 > 1
  - 当前出牌人选择Pass（不出）
  - 当前出牌人不是turn（即响应者/接力阶段的非出牌人）
- **用户提示**：当★输入被约束过滤的单张时，CLI 显示差异化错误：
  `"❌ 无效出牌：因为下家只剩一张牌，你必须出最大的单张 X（而非 Y），请重新选择。"`

### 全局最大牌型直接接管
- **定义**：一手出牌在所有玩家手牌中，是该牌型的**全局唯一最大**
- **判断标准**：
  - 单张：该牌 order 大于所有玩家手牌中的最大 order
  - 对子/三条/四条：该 (rank, top_suit) 组合是所有玩家中该牌型能形成的最大且唯一
- **触发效果**：
  - turn 保持在当前出牌人（不后移）
  - **trick 清空为 None**（关键！出牌人可以重新自由选择任意牌型）
  - 打印 `"🎯 全局最大！直接继续出牌"`
- **实现函数**：`is_global_max(move, all_masks) -> bool`
  - 兼容自由出牌 (5-tuple) 和接力压制 (3-tuple) 两种 move 格式
  - 忽略 mask=0 的空手玩家
  - 唯一性检查：多个玩家有相同最大牌型时不触发

## 牌权传递逻辑
- 出牌后：turn移到下家 (turn+1) % 5
- **例外：全局最大** → turn 不后移，trick 清空，当前玩家继续出牌
- 如果下家管不上：Pass → turn继续移到下下家
- 回到出牌人时检查：
  - 若一圈无人管（所有人Pass）→ 清空trick为None，出牌人继续出
  - 若中间有人出了牌 → 继续正常轮转

---

## 抢A规则
- 亮♦A → 顺时针首个"抢"生效
- 换手：bidder += ♦A，原主 -= ♦A（6/4/5张）
- 抢A局结算：★第1 → +10×(N-1)；★输 → -12×(N-1)给每人
- 无人抢A → 游戏结束，退出训练模式

---

## 游戏流程
游戏支持两种训练模式：

### 训练模式一（实时交互训练）
**目标**：通过实时交互，帮助★玩家学习"怎么出能赢"，每步获得胜败反馈。

**流程**：
1. 发牌：洗牌后每个玩家发5张（5人局共25张）
2. 抢A阶段：从♦A持有者开始顺时针轮询，首个输入y的玩家成为★（抢到A），♦A交换（bidder+1张，原主-1张，形成6/4/5张分布）
3. 初始化：★移到索引0（先手），其余玩家保持相对顺序；GameState初始化（turn=bidder，trick=None）
4. 游戏循环：
   - **★回合**：
     - 调用 `solve(state)` 分析当前局面，显示 `📊 局面分析：✅ ★必胜` 或 `❌ ★必败`
     - 枚举所有合法出牌，标注 `✅ 胜招`（该出牌后★仍有必胜路径）或 `❌ 败招`（该出牌后★必败）
     - 玩家输入order选择出牌（支持单张或多张，如"0"或"0 1"）
     - 输入验证：非法出牌时显示错误原因（如下家只剩一张牌时的约束提示）
   - **对手回合**：
     - AI调用 `solve(state)`，选择"最恶毒"的合法出牌（优先选让★输的move；若无则随机选）
     - 打印对手出牌及剩余手牌
   - **全局最大接管**：若玩家出牌是全局最大牌型，打印 `🎯 全局最大！直接继续出牌`，turn不后移，trick清空
   - **Pass处理**：玩家管不上时选择不出，显示剩余手牌，turn移到下家
5. **终局**：任一玩家手牌为空 → 游戏结束
   - ★先出完 → 打印 `🎉 ★ 获胜！`
   - 对手先出完 → 打印 `💀 玩家X先出完，★失败！`

**实现要点**：
- 每步调用 `solve(state)` 实时分析胜败
- 合法出牌枚举时标注胜败（通过 `analyze_moves(state)` 分类）
- 下家只剩一张牌时，单张出牌必须出最大，输入错误时显示差异化提示
- 全局最大牌型触发直接接管（trick清空，turn不变）

---

### 训练模式二（同盟必胜序列分析）
**目标**：基于上帝视角（全手牌可见），分析同盟是否存在必胜策略，帮助★玩家理解"怎么出会输"以及对手的最优反制路径。

**触发时机**：抢A完成 → 换手牌 → 初始化GameState后

#### 静态分析阶段（★出牌前）—— 多♦A出牌分支验证
- **输入**：初始GameState（turn=0，trick=None）
- **处理**：调用 `verify_all_da_moves(state)` 验证★的每种含♦A出牌，同盟是否都有必胜序列
  - 逐一枚举★手牌中所有含♦A的合法出牌（通过 `enumerate_da_moves`）
  - 对每种出牌：应用出牌 → 处理全局最大接管 → 调用 `find_winning_sequence` 搜索同盟必胜序列
- **输出**（同盟全有必胜序列时）：
  ```
  ⚠️ 当前局面同盟有必胜策略！
  ★ 可选择以下含♦A出牌，同盟均有必胜应对：
  【出法1】单张: ♦A
  同盟必胜序列：
    ★ → 出 ♦A（单张）
    玩家1 → 出 ♣A（单张）
    ★ 最终无法出牌/最后出完
  【出法2】对子: ♦A ♣A
  同盟必胜序列：
    ...
  ```
- **输出**（存在★胜招时）：
  ```
  ⚠️ ★ 有胜招！
  【出法1】单张: ♦A → 同盟无必胜序列 ❌
  【出法2】对子: ♦A ♣A → 同盟必胜 ✅
  建议：出单张 ♦A 可破局！
  ```
  - 若存在★胜招（任意含♦A出牌同盟无必胜序列），训练模式二直接退出

#### 动态分析阶段（★出牌后）—— 强制执行模式
- **触发**：★每出一手牌后（仅当所有含♦A出牌同盟都有必胜序列时进入）
- **输入**：★出牌后的新GameState
- **强制序列追踪**：维护 `current_sequence` 列表追踪当前同盟必胜路径
  - 初始序列：`find_best_response(state)`（★出牌前预计算）
  - ★每步出牌后重新调用 `find_best_response(state)` 更新序列
  - 全局最大接管时清空并重置序列
- **对手回合强制执行**：
  - 遍历 `current_sequence` 查找当前 `state.turn` 对应的预计算出牌
  - 找到 → 调用 `execute_forced_move(state, forced_move)` 强制出牌并从序列移除
  - 未找到（如 Pass 场景）→ 降级使用 `play_turn(state)` 正常出牌
- **输出**：
  - 当前状态下同盟的最优出牌序列（最恶毒路径）
  - 格式示例：
    ```
    ★ 出: ♦A（单张）
    ⚠️ 同盟有最优应对策略！
    同盟最优应对序列：
    玩家1 → 出 ♣A（单张）
    ★ 最终失败！
    🤖 [强制] 玩家1 出: ♣A（单张）
    ```
  - 若无必胜序列：`✅ 此手仍在必胜域内，同盟暂无必胜策略`

**实现函数**：
- `enumerate_da_moves(mask) -> List[move]`：枚举★手牌中所有包含♦A（order=0）的合法出牌（5-tuple列表）
- `verify_all_da_moves(state) -> Dict[str, List|None]`：验证★每种含♦A出牌后同盟是否有必胜序列
- `format_multi_da_verification(result_dict) -> str`：格式化多分支验证结果为可读字符串
- `find_winning_sequence(state) -> List[Action] | None`：搜索一条同盟必胜序列（DFS沿让★输分支，含全局最大接管、Pass穿越、深度保护）
- `find_best_response(state) -> List[Action]`：搜索当前状态下同盟最优应对序列（最恶毒路径）
- `format_sequence(seq) -> str`：格式化同盟必胜序列为可读字符串
- `format_best_response(seq) -> str`：格式化同盟最优应对序列为可读字符串
- `execute_forced_move(state, forced_move) -> GameState`：强制当前玩家按指定move出牌，含全局最大接管处理、出牌日志打印、剩余手牌显示

**与训练模式一的区别**：
| 维度 | 训练模式一 | 训练模式二 |
|------|-----------|-----------|
| 核心功能 | 实时交互，每步标注✅/❌ | 赛前多分支分析 + 赛中最优应对序列 |
| 分析时机 | 每步出牌后 | 抢A后（静态多分支验证）+ 每步出牌后（动态） |
| 输出内容 | 胜败标注 + 可选胜招 | 每种含♦A出牌的同盟必胜/无必胜判定 + 最优出牌序列 |
| 目标 | 帮玩家"怎么出能赢" | 帮玩家"怎么出会输 + 对手怎么打" |

## 预设手牌场景加载

支持通过配置文件加载预设手牌，替代随机发牌。

### 启动方式
```bash
# 加载配置文件，交互选择场景
python main.py --load configs/hands.yaml

# 直接指定场景ID（跳过交互）
python main.py --load configs/hands.yaml --scene 1

# 不传 --load → 随机发牌（向后兼容）
python main.py
```

### YAML 配置文件格式
- 一个文件支持多个场景（`scenarios` 列表）
- `bidder: null` → 自动检测 ♦A 持有者作为★
- `bidder: 0~4` → 指定★（若非♦A持有者，自动执行换手）
- 花色：D=♦, C=♣, H=♥, S=♠（大小写不敏感）
- 牌面：A/2~10/J/Q/K

### 验证规则
| 验证项 | 规则 | 违规示例 |
|--------|------|----------|
| 牌合法性 | order 必须在 0~24 | 非法花色/牌面/order超范围 |
| 无重复牌 | 同一order不能出现在多个玩家 | 玩家0和玩家1都有 D A |
| 手牌总数 | = 25（5玩家×5张） | 总数≠25 |
| ♦A 存在性 | `bidder:null` 时必须存在 D A | bidder=null 但无♦A |
| 玩家数量 | 固定5人，players 含 0~4 键 | 缺少某玩家或玩家数≠5 |

### 加载流程
1. 解析 YAML → 场景选择 → 手牌验证 → Card 解析
2. 跳过随机发牌、跳过抢A轮询
3. 直接确定★并执行换手（若需要）
4. 进入正常游戏循环（训练模式一/二）

## 实现细节
- 手牌使用 25-bit mask 表示（每个bit对应一张牌的order）
- 使用 memoized DFS（@lru_cache on solve）求解必胜/必败
  - ★回合：存在一条路径通向 True → True（★必胜）
  - 对手回合：所有路径都通向 True → True（对手选择最恶毒的会阻止★）
- 输入容错：抢A阶段大写Y/小写y均视为抢
- 首出♦A验证：trick=None且♦A在手时，必须包含order=0
- 牌型中文名：单张/对子/三条/四条
- 约束提示：下家只剩一张牌时输入非最大单张 → 显示差异化错误原因
- 全局最大接管：trick清空为None（非保持），turn不后移
- 训练模式一：实时交互，每步分析胜败并标注
- 训练模式二：静态多♦A分支验证 + 动态强制执行对手出牌
- 多♦A分支验证：enumerate_da_moves 筛选含order=0的自由出牌；verify_all_da_moves 对每种含♦A出牌逐一验证同盟必胜序列；format_multi_da_verification 区分"同盟全必胜"与"★有胜招"两种输出
- 同盟必胜序列DFS：_dfs_win_seq 沿"让★输"分支搜索，含深度保护(_MAX_SEQ_DEPTH=200)、Pass穿越、全局最大接管处理

## 文件结构
- `models.py` — 数据模型（Card/Trick/GameState）、常量（_TYPE_CN/牌型层级）、格式化工具
- `deck.py` — 牌组构建、洗牌发牌、抢A换手、mask初始化
- `moves.py` — 牌型识别（get_counts）、合法出牌枚举（get_legal_moves_free / get_legal_moves_follow）、全局最大判定（is_global_max）、含♦A出牌枚举（enumerate_da_moves）
- `solver.py` — 求解器（solve / analyze_moves）、局面推进（_apply_move / advance_turn / check_terminal）
- `sequence.py` — 同盟序列分析（find_winning_sequence / find_best_response / verify_all_da_moves 及对应format函数）
- `cli.py` — CLI交互（play_turn）、强制执行（execute_forced_move）、对手AI出牌
- `config_loader.py` — YAML配置文件加载、手牌验证（重复牌/牌数/♦A存在性）、Card解析与场景选择
- `configs/hands.yaml` — 示例预设手牌场景集（2个场景：★极好/★必败）
- `game.py` — 向后兼容的聚合导出入口
- `main.py` — 演示入口（训练模式一/二；命令行参数：`--load`、`--scene`、`--test-best-response`、`--test-multi-da`）
- `test_game.py` — 单元测试（牌型识别、枚举器、求解器、抢A、牌权传递、Pass、下家约束、全局最大、同盟序列、边界条件）

### 关键设计决策
- **手牌卡片固定大小**：`.card-hand-area .poker-card { width: 3.2rem; height: 4.8rem }`，通过 `margin: auto` 居中，防止 st.columns 导致的横向拉伸
- **透明按钮叠加**：st.button 高度 8px 透明 → 负 margin 卡片覆盖 → `pointer-events: none` 穿透点击
- **选中态**：inline style `translateY(-8px) scale(1.08)` + 蓝色边框 + 阴影，`z-index: 10` 浮于其他牌之上
- **历史出牌**：`all_trick_history` 跨轮保留，按轮次分组显示，`max-height: 360px` 可滚动
- **轮次追踪**：`current_round` 在每轮开始时（trick 从非 None → None）递增
- **★ 出牌控件上移**：放在「历史出牌」expander 和「你的手牌」之间，对手回合控件在 `render_interaction()` 中独立显示
